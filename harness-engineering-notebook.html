01

What Is Harness Engineering?

🟢 Beginner Level

Imagine you've just bought a racehorse — arguably the most powerful animal on earth. It's fast, it's smart, and it has incredible instincts. But left to itself, it will wander wherever it pleases. To turn it into a race winner, you need a saddle, reins, a bridle, and a jockey who knows how to use them. That entire system is the harness.

🐎 The Core Analogy

The AI model (Claude, GPT-4, Gemini) is the horse — raw, powerful intelligence. Harness engineering is everything else: the reins that steer, the bit that constrains, the saddle that connects, and the training regimen that makes it reliable. The horse provides the power. The harness provides the control.

More technically: harness engineering is the discipline of designing and maintaining the control systems that govern how an AI agent perceives its environment, selects actions, and validates outputs. It covers everything that wraps the model — but not the model itself.

🚫

What harness engineering is NOT

It's not prompt engineering (tuning what you say). It's not model training or fine-tuning. It's not ML research. It's the infrastructure layer in between.

✅

What harness engineering IS

Building the guides, sensors, tools, feedback loops, and constraints that turn a raw LLM into a reliable, autonomous, production-grade AI agent.

💡

The OS analogy

Model = CPU. Context window = RAM. Harness = Operating System. Agent = Application. You wouldn't run software directly on a CPU — same logic applies here.

A Precise Definition

Harness engineering is the practice of designing environments, constraints, feedback loops, and infrastructure that make AI agents reliable at scale. A harness engineer doesn't build the model — they build everything around it: the context it receives, the tools it can access, the guardrails that prevent mistakes, the verification systems that catch errors, and the recovery mechanisms that handle failures.

"If you're not the model, you're the harness."

— Vivek Trivedy, LangChain (March 2026)

Why Does It Matter Now?

Three converging trends made harness engineering essential in 2026.

01

Models commoditized

Claude, GPT-4, Gemini, and open-source models now perform within a narrow band on benchmarks. The model is no longer the competitive advantage — the system around it is.

02

Agents went to production

In 2025 most agents were demos. In 2026, organizations deploy agents that write production code, manage infrastructure, and make financial decisions. The bar is now "can't go down."

03

Benchmarks stopped mattering

Standard benchmarks measure single-turn tasks. Production agents run for hours, encounter timeouts, context limits, and tool failures. A 1% benchmark gain means nothing if the agent drifts at step 50.

📊 Key Statistic

27% of AI agent failures trace to data quality and context issues — not model limitations, not architecture. This is why harness engineering, especially context engineering, is now considered the most critical AI skill of 2026.

History & Origins

1940s – 1990s
Electrical wire harnesses
The word "harness" came from electrical engineering — bundling wires into organized, insulated assemblies. Vehicle wiring harnesses connected engines, dashboards, and sensors reliably. The metaphor: organized structure enabling complex systems.
2000s – 2020s
Software test harnesses
In software, the term moved to testing. A test harness provided a controlled environment with mock objects, test data, and test runners to isolate and verify code behavior — JUnit, pytest, etc.
Early 2026
AI harness engineering coined
OpenAI revealed that their internal product — built with over 1 million lines of code — had zero human-written lines. Engineers didn't write code; they designed the system that let AI agents write code reliably. That system is the harness.
2026 (Now)
An established discipline
Harness engineering is now a recognized specialty. Mitchell Hashimoto popularized the formula "Agent = Model + Harness." Anthropic, LangChain, Martin Fowler, and OpenAI have all published canonical guides on the discipline.

The Three Pillars

Every harness engineering effort maps to three interconnected disciplines.

🟢 Foundation Knowledge
📚 Pillar 1

Context engineering is the practice of managing what information an AI agent can access, when it accesses it, and how that information is structured. It directly determines whether an agent can maintain coherent behavior across extended tasks.

Every AI model has a limited context window. Even models with 200,000-token windows face practical limits when agents run multi-hour tasks. The context fills up, information gets pushed out, and the agent "forgets" earlier decisions.

Context Engineering Techniques

Context compression

Reducing the size of information in the agent's working memory while preserving what's relevant to the current task. The agent can keep running without losing critical state.

Dynamic context injection

Loading the right information at the right time, rather than stuffing everything into the initial prompt. Think of it like paging memory in an OS — only bring in what you need, when you need it.

Knowledge persistence

Storing decisions, progress, and state outside the context window (in files, databases, or structured logs) so the agent can recover across sessions. A progress.txt file is a classic pattern.

Context resets vs. compaction

Compaction summarizes earlier conversation in-place. Resets start a fresh agent with a structured handoff artifact. Resets solve "context anxiety" — where agents wrap up work prematurely as they approach perceived context limits.

progress.txt pattern
# Agent session handoff artifact

🔒 Pillar 2

Architectural constraints are about preventing the agent from doing the wrong things. Production AI agents need boundaries. Without them, an agent tasked with refactoring code might rewrite the entire codebase — or worse.

The key insight: constraints don't reduce agent capability — they increase it. An agent operating within well-defined boundaries can act with more autonomy because the harness catches mistakes. Without constraints, you need constant human oversight, which defeats the purpose of agents.

Types of Constraints

Tool access controls

Define which tools an agent can use, which files it can modify, and which operations require human approval. Never give an agent more power than it needs for its current task.

Structural enforcement

Use linters, type checkers, and CI/CD pipelines to automatically verify that agent output meets quality standards. Deterministic checks are your first line of defense.

Scope boundaries

Limit what the agent can change in a single task to prevent cascading modifications. Like database transactions — atomic, bounded, reversible.

Safety guardrails

Filter agent outputs for security vulnerabilities, harmful content, or policy violations before they reach production. Both rule-based and LLM-as-judge approaches apply here.

AGENTS.md constraint example
# Scope Constraints
You MUST NOT:
  - Modify files outside /src/ and /tests/
  - Delete any file without explicit user confirmation
  - Make database schema changes without running migration scripts
  - Deploy to production without passing all tests

You MUST:
  - Run npm test after any code change
  - Check existing tests before adding new code
  - Write a commit message for each logical change
🌀 Pillar 3

Entropy management addresses the slow decay of AI-generated codebases. Over time, AI-generated code accumulates inconsistencies: variable naming drifts, documentation falls out of sync, dead code accumulates, test coverage drops. This is entropy.

In traditional software development, humans notice and correct entropy naturally during code review. But when agents generate code at high speed, entropy accumulates much faster. Left unmanaged, it compounds into a system no agent can reliably work with.

Entropy Management Strategies

Scheduled entropy passes

Periodically run a dedicated "cleanup agent" that finds and fixes inconsistencies — dead code, stale comments, naming drift — before they compound.

Codemods & structural transforms

Use deterministic tools (OpenRewrite, jscodeshift, etc.) to enforce naming conventions and structural patterns across the entire codebase automatically.

Continuous documentation sync

Run a documentation agent on each merge that checks whether code changes invalidate existing docs — and updates them if so. Never let docs drift.

Architecture fitness tests

ArchUnit-style tests that verify structural constraints (module boundaries, dependency rules) on every commit — turning architectural rules into executable tests.

Anatomy of an Agent Harness

The Full Harness Stack

The Filesystem: Most Fundamental Harness Primitive

Of all harness components, the filesystem is arguably the most foundational. It unlocks: agents get a workspace to read data, code, and docs; work can be incrementally added and offloaded instead of holding everything in context; and multiple agents and humans can coordinate through shared files.

Minimal harness structure (Python)
from anthropic import Anthropic
import subprocess, json

client = Anthropic()

def run_harness(task: str, workspace: str):
    # Load context from persistent state
    state = load_state(workspace)
    
    # Build context-engineered system prompt
    system = build_system_prompt(state)
    
    while not state["complete"]:
        response = client.messages.create(
            model="claude-sonnet-4-20250514",
            system=system,
            messages=[{"role": "user", "content": task}],
            tools=get_tools()
        )
        
        # Execute tool calls from agent
        results = execute_tools(response.content)
        
        # Run sensors (linters, tests)
        feedback = run_sensors(workspace)
        
        # Update persistent state
        save_state(workspace, state, feedback)
        
        # Check context budget — reset if needed
        if context_near_limit(response):
            system = reset_with_handoff(state)

Guides & Sensors

🟡 Intermediate Level

The two control dimensions of any harness: feedforward and feedback.

From Martin Fowler's canonical article on harness engineering, the key framework is feedforward vs. feedback controls:

→

Guides (Feedforward)

Anticipate the agent's behavior and aim to steer it before it acts. Guides increase the probability of good results on the first attempt.

Examples: AGENTS.md, Skills files, reference docs, coding conventions, bootstrap scripts, codemods.

↺

Sensors (Feedback)

Observe after the agent acts and help it self-correct. Particularly powerful when they produce signals optimized for LLM consumption.

Examples: linters, test runners, ArchUnit, LLM review agents, browser screenshots, custom error messages with correction hints.

Computational vs. Inferential Controls

TypeDirectionSpeedDeterministic?Example
Computational Either Milliseconds ✅ Yes eslint, pytest, mypy, ArchUnit
Inferential Either Seconds–minutes ⚠️ Probabilistic LLM code review, AI design grader
💡 The Power of the Feedback Loop

A harness with only guides will encode rules but never learn if they worked. A harness with only sensors will catch problems but never prevent them. You need both — guides increase first-attempt quality, sensors create the self-correction loop that improves over time. Separately, each is weak. Together, they compound.

Example: Custom Linter Message as "Positive Prompt Injection"

sensor output optimized for LLM consumption
// ESLint custom rule message — written for the agent, not the developer
{
  "ruleId": "no-direct-db-from-controller",
  "severity": 2,
  "message": "Direct DB calls in controllers violate our 
    3-layer architecture. Move this to a repository class in 
    /src/repositories/. See /docs/architecture.md#layer-rules 
    for the pattern to follow.",
  "line": 42
}
// The message tells the agent EXACTLY what to do next — 
// not just what went wrong.

Harness Architecture Patterns

Pattern 1

Single agent with harness — the simplest pattern. One model, one context window, with guides and sensors wrapping it. Suitable for bounded, well-defined tasks under ~30 minutes.

Pattern 2 — GAN-inspired

Generator–Evaluator: Two separate agents play adversarial roles. The generator produces output; the evaluator critiques it and sends feedback. Inspired by Generative Adversarial Networks. Used by Anthropic for frontend design quality.

Key insight from Anthropic's research: tuning a standalone evaluator to be skeptical is far more tractable than making a generator critical of its own work. The generator produces; the evaluator judges. Separating them breaks the self-assessment bias.

Pattern 3 — Production Grade

Planner → Generator → Evaluator: Three-agent architecture for long-running tasks. The planner decomposes work into chunks, generators implement each chunk, evaluators validate the result. Used for multi-hour autonomous coding sessions.

Multi-Agent Harnesses

🟡 Intermediate Level

As tasks become more complex, a single agent can't hold all necessary context or perform all necessary functions. Multi-agent harnesses distribute work across specialized agents that coordinate through shared state.

Why spawn subagents?

Parallel execution

Independent subtasks run simultaneously in isolated contexts. A research harness might spawn 5 agents searching different sources in parallel — 5× faster.

Context isolation

Each subagent gets only the context it needs. Prevents cross-contamination and "context anxiety." Specialized agents perform better within their domain.

Role specialization

Agents can be tuned for specific functions — a planning agent uses different system prompts and tools than a coding agent or a security-review agent.

Coordination Patterns

PatternHow agents coordinateBest for
File-based handoffAgent A writes structured artifact; Agent B reads and continuesLong-running coding sessions, session resets
Orchestrator → workersCentral agent routes tasks to specialized agents, collects resultsResearch pipelines, data processing
Peer review loopAgent A creates, Agent B critiques, Agent A revisesContent quality, code review, design iteration
Streaming fan-outOrchestrator decomposes → N agents run in parallel → merge resultsCompetitive analysis, parallel search, batch jobs
Subagent spawning (Claude Agent SDK)
import { Agent, AgentSDK } from '@anthropic/agent-sdk';

const orchestrator = new Agent({
  model: 'claude-sonnet-4-20250514',
  system: `You are a research orchestrator. 
    Given a topic, spawn 3 specialized subagents to:
    1. Search recent papers
    2. Find industry examples  
    3. Compile a synthesis`
});

const subagents = await orchestrator.spawnAll([
  { role: 'paper-searcher', tools: ['web_search'] },
  { role: 'industry-finder', tools: ['web_search'] },
  { role: 'synthesizer', tools: ['read_file', 'write_file'] }
]);

const results = await Promise.all(subagents.map(a => a.run()));

Context Engineering Deep Dive

Context engineering is so foundational that it deserves its own section. The relationship between context engineering and harness engineering is that context engineering provides the means to make guides and sensors available to the agent.

The Context Budget Problem

Context anxiety

Models approaching their perceived context limit begin wrapping up work prematurely — even when tokens remain. This is a harness problem, not a model problem. Solution: context resets with handoff artifacts.

Compaction vs. Resets

Compaction summarizes earlier conversation in-place — preserves continuity but doesn't eliminate anxiety. Resets provide a clean slate with structured handoff — eliminates anxiety at the cost of orchestration overhead.

Dynamic injection

Don't front-load all context. Inject relevant documentation, reference files, and state at the exact moment the agent needs them. Reduces wasted tokens and keeps relevance high.

Entropy Management

🔴 Advanced Level

Entropy is the silent killer of AI-generated codebases. When humans write code, they naturally maintain a mental model of the whole system and correct drift during reviews. When agents write code at high speed, this natural correction mechanism is absent.

🔬 Scientific Analogy

In thermodynamics, entropy always increases in a closed system. An AI-generated codebase without entropy management is a closed system — naming conventions drift, docs fall out of sync, dead code accumulates. Your entropy management strategy is the energy input that keeps the system ordered.

Signs of Entropy in AI-Generated Code

  • 🔴Same concept has 3 different variable names across different files
  • 🔴Functions that are called nowhere but never deleted
  • 🔴Documentation that describes yesterday's implementation
  • 🔴Test coverage dropping as new code is added without tests
  • 🔴Import chains that bypass architectural boundaries
  • 🔴README that was accurate on day 1 but hasn't been updated since

Entropy Management Harness

entropy-agent.py — scheduled cleanup agent
# Run weekly or on merge to main
entropy_system = """
You are an entropy management agent. Your job is NOT to build 
features — it is to find and fix inconsistencies.

Check for:
1. Dead code (functions, variables, imports never referenced)
2. Naming drift (same concept, different names across files)
3. Stale documentation (code changed but docs didn't)
4. Missing tests (new functions without test coverage)
5. Architecture violations (imports crossing module boundaries)

For each issue: fix it silently. Don't add features. 
Don't refactor logic. Only clean what's already broken.
Report what you fixed in /docs/entropy-log.md.
"""

run_agent(
    system=entropy_system,
    tools=["read_file", "write_file", "bash"],
    max_steps=100,
    workspace="./src"
)

Evaluation Strategies

One of the hardest problems in harness engineering: how do you know if the agent did a good job? Especially for subjective tasks like design or writing, there's no binary pass/fail.

The Self-Evaluation Problem

⚠️ Key Insight

When asked to evaluate work they've produced, agents tend to respond by confidently praising the work — even when quality is obviously mediocre. This bias is systematic and reliable. Separating the agent doing the work from the agent judging it is the core solution. Tuning a standalone evaluator to be skeptical is far more tractable than making a generator self-critical.

Grading Criteria Design

From Anthropic's frontend design research, the key insight: while aesthetics can't be fully reduced to a score, they can be graded against concrete criteria. "Is this beautiful?" is unanswerable. "Does this follow our design principles?" is gradable.

Evaluator grading criteria (Anthropic's frontend design harness)
GRADING CRITERIA — score each 1–10:

Design Quality (weight: 3x)
  Does the design feel like a coherent whole? Do colors, 
  typography, layout, and imagery combine into a distinct 
  mood and identity? Or is it a collection of parts?

Originality (weight: 3x)  
  Are there custom decisions? Or template layouts, library 
  defaults, and AI-generated patterns? A human designer 
  should recognize deliberate creative choices.
  FAIL: purple gradients on white cards, generic stock components.

Craft (weight: 1x)
  Technical execution: typography hierarchy, spacing consistency,
  color harmony, contrast ratios.

Functionality (weight: 1x)
  Can users find primary actions and complete tasks?

After scoring each criterion, write a concrete critique and 
list EXACTLY what the generator should change in the next pass.

Long-Running Agent Harnesses

🔴 Advanced / Production

Long-running tasks (30 minutes to several hours) require special harness design. Standard single-session approaches fail at scale. Here's the complete architecture from Anthropic's production work.

The Two Failure Modes

Context window exhaustion

Models lose coherence as context fills. They begin summarizing prematurely, skipping work, or producing repetitive outputs. Addressed by: context resets with handoff artifacts.

Self-evaluation leniency

Agents confidently praise mediocre work when asked to self-assess. Pronounced in subjective tasks. Addressed by: separate evaluator agent with adversarial calibration.

Context Reset Architecture

Planner creates task list

Decomposes the full project into tractable chunks (features, modules, or sessions). Writes structured task list to a persistent artifact file.

Generator implements chunk N

Works within a bounded context window on a single chunk. Uses the task list and previous session's handoff artifact for context. Runs until chunk complete OR context threshold hit.

Handoff artifact written

Before context reset, the agent writes a structured summary: what was done, what's next, known issues, file locations. This is the memory that survives the reset.

Context reset + new agent

Fresh agent spawned. Reads handoff artifact. Continues seamlessly. The clean slate eliminates context anxiety.

Evaluator scores output

After each chunk (or each session), the evaluator agent reviews the output, runs tests, checks quality criteria, and produces critique for the next generator.

🏗️ Real-World Result

Using this three-agent architecture (planner + generator + evaluator with context resets), Anthropic's Labs team produced rich full-stack applications over multi-hour autonomous coding sessions — including a Dutch art museum website that, on the 10th evaluation cycle, spontaneously reimagined itself as a 3D spatial experience with CSS perspective-rendered gallery rooms.

Case Studies

CASE STUDY 01 — OPENAI

1,000,000 lines of code. Zero written by humans.

The Setup

OpenAI's internal product team revealed that their production system — over one million lines of code — was built entirely by AI agents. Human engineers designed the harness, not the code. This announcement effectively coined the term "harness engineering" as a discipline.

Key Harness Design Decisions
  • AI agents AND deterministic linters used in tandem
  • Deterministic checks catch structural violations instantly
  • AI agents catch subtler semantic issues (naming drift, doc sync)
  • Human role: steering the harness, not writing code
CASE STUDY 02 — ANTHROPIC LABS

GAN-Inspired Frontend Design: From Generic to Museum-Quality

The Problem

Without intervention, Claude produces technically functional but visually unremarkable frontends — "safe, predictable layouts." The challenge: how do you make subjective quality gradable and improvable?

Solution: a Generator–Evaluator harness inspired by Generative Adversarial Networks.

The Harness
  • Generator creates HTML/CSS/JS frontend
  • Evaluator uses Playwright to navigate the live page, screenshots it, then scores against 4 criteria
  • Feedback flows back to generator for 5–15 iterations
  • Result: a 3D CSS-perspective gallery room spontaneously emerged on iteration 10
CASE STUDY 03 — THOUGHTWORKS

Coding Agent Harness for Enterprise Software Delivery

The Context

Birgitta Böckeler at Thoughtworks applied harness engineering to enterprise coding agents — where the challenge is not just getting good code, but maintaining trust and quality across an entire delivery organization.

The Approach (Concentric Harnesses)
  • Inner harness: coding agent's built-in system (tools, orchestration)
  • Outer harness: user-built guides + sensors specific to their codebase
  • Guides: AGENTS.md, project-specific skills, architecture docs
  • Sensors: ArchUnit tests for module boundaries, custom linter messages written FOR the agent to read

Knowledge Check

Test Your Harness Engineering Understanding

1. An agent's context window is filling up and it's starting to wrap up work prematurely. What harness technique addresses this?
✅ Correct! Context resets provide a clean slate for the agent, eliminating "context anxiety." The key is the handoff artifact — a structured file carrying state across the reset.
❌ Not quite. A larger model delays the problem but doesn't solve it. Context anxiety is a behavior pattern that emerges regardless of window size as the context fills. The solution is architectural: context resets with handoff artifacts.
2. You ask an agent to grade a frontend design it just created. The agent gives it 9/10 and says it's excellent. But to you it looks generic and bland. This is called:
✅ Correct! Self-evaluation leniency is systematic — agents reliably grade their own work generously. The harness solution: separate generator and evaluator agents, calibrating the evaluator to be skeptical.
❌ This is self-evaluation leniency bias — a systematic tendency for agents to generously grade their own output. It's solved by separating generator and evaluator agents in the harness.
3. In the feedforward/feedback framework, which of the following is a FEEDFORWARD (guide) control?
✅ Correct! AGENTS.md steers the agent BEFORE it acts — that's feedforward. Linters, test suites, and LLM judges all run AFTER the agent acts — those are feedback (sensor) controls.
❌ AGENTS.md is the answer. It's a feedforward guide — it steers the agent before it acts. Linters, test suites, and LLM judges are all feedback sensors — they observe after the fact.
4. In the formula Agent = Model + Harness, if you swap the model from GPT-4 to Claude, what should happen to the harness?
✅ Correct! A well-designed harness is largely model-agnostic — tools, filesystem, sensors, and architecture patterns transfer. You'd tune guides (system prompts, skills files) for the new model's characteristics, but the infrastructure stays.
❌ A well-designed harness transfers across models. The infrastructure (tools, filesystem, sensors, orchestration) is model-agnostic. You'd tune the guides (system prompts, skills) for model-specific behaviors, but you don't start over.

Glossary

Agent
Model + Harness — the complete system capable of autonomous task execution.
AGENTS.md
A markdown file of standing instructions for an AI coding agent — the primary guide mechanism.
Architectural constraint
A rule enforced by the harness that limits what the agent can do — tool access, file permissions, scope boundaries.
Compaction
Summarizing earlier conversation in-place to free context space, preserving agent continuity.
Context anxiety
A behavior pattern where agents wrap up work prematurely as they approach their perceived context limit.
Context engineering
The practice of managing what information an agent can access, when, and how it's structured.
Context reset
Clearing the context window and starting a fresh agent, passing state via a handoff artifact.
Entropy management
Strategies for preventing and correcting the slow degradation of AI-generated codebases over time.
Evaluator agent
A separate agent whose sole job is to score and critique the output of a generator agent.
Feedforward (guide)
A harness control that steers the agent before it acts — preventing problems rather than correcting them.
Feedback (sensor)
A harness control that observes after the agent acts and provides signals for self-correction.
Handoff artifact
A structured file written at the end of one agent session, providing the state needed for the next session.
Harness
Everything that wraps the model: guides, sensors, tools, orchestration, constraints, and memory. Agent minus Model.
Harness engineering
The discipline of designing the systems, constraints, and feedback loops that make AI agents reliable in production.
Harnessability
How easy a codebase or task is to wrap in a harness — a property you can engineer into your system.
LLM-as-judge
Using a language model as an evaluation sensor to assess output quality, code correctness, or design quality.
MCP (Model Context Protocol)
A standard for connecting AI models to external tools and data sources — a harness tool layer protocol.
Self-evaluation leniency bias
The systematic tendency of agents to generously grade their own output, regardless of actual quality.
Skill
A reusable guide document (like a SKILL.md) that encodes best practices for a specific task domain.
Steering loop
The human's role in harness engineering — iterating on guides and sensors based on observed failure patterns.
